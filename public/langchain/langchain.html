Module 0 · Lesson 1
What is LangChain?
The mental model that makes everything click. Before touching a single line of code, you need to understand what problem LangChain solves and why it exists.
📖 Concept lesson
⏱ ~10 min
🎯 Beginner
1 The Problem LangChain Solves

You've been building MyApp directly with the Claude API, ChromaDB, Voyage AI, BM25, and FastAPI — and you've probably noticed that gluing all these pieces together takes a lot of boilerplate. Every project ends up with the same patterns: format a prompt, call a model, parse the output, pass it to a retriever, chain it with another call...

LangChain is a framework that standardises these patterns. Think of it as the "Spring Boot of LLM apps" — just like Spring Boot abstracts away servlet configuration and dependency injection for Java, LangChain abstracts away the plumbing of LLM-powered applications.

💡
The Core Insight
LangChain doesn't do anything new. It just gives common names and interfaces to the patterns you're already using. A "chain" is just a sequence of operations. A "retriever" is just something that returns documents. A "tool" is just a function an agent can call. Once you see this, the whole library demystifies itself.
2 Raw API vs LangChain — Side by Side

Let's look at a simple prompt-and-respond flow done both ways. This is the clearest illustration of what LangChain buys you.

✗ WITHOUT LANGCHAIN
python 
import anthropic

client = anthropic.Anthropic()

# Build the prompt manually every time
def ask_about_record(record_type: str):
    prompt = f"Explain {record_type} your domain in simple terms."
    
    response = client.messages.create(
        model="claude-3-5-sonnet-20241022",
        max_tokens=1024,
        messages=[{"role": "user", 
                   "content": prompt}]
    )
    # Manual extraction
    return response.content[0].text
✓ WITH LANGCHAIN
python 
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# Declare once, reuse infinitely
chain = (
    ChatPromptTemplate.from_template(
        "Explain {record_type} your domain in simple terms."
    )
    | ChatAnthropic(model="claude-3-5-sonnet-20241022")
    | StrOutputParser()
)

result = chain.invoke({"record_type": "life"})

Notice the LangChain version is declarative — you describe what to do, not how. The | pipe operator chains operations together. This is LCEL (LangChain Expression Language), and it's the backbone of everything.

3 The 5 Core Abstractions

Everything in LangChain maps to one of five ideas. Memorise these — every class, every function in the library fits one of these buckets.

🔷 Models

Wrappers for LLMs and chat models. ChatAnthropic, ChatOpenAI, ChatGoogleGenerativeAI. Standardises the call interface across providers.

📝 Prompts

Templates with variables: PromptTemplate, ChatPromptTemplate. Separates prompt structure from runtime values.

📤 Output Parsers

Transform raw LLM output into structured data. StrOutputParser, JsonOutputParser, PydanticOutputParser.

📚 Retrievers

Abstracts any source that returns documents — vector stores, BM25, web search, SQL. Everything speaks a common interface.

🤖 Agents

LLMs that decide which tools to call, in what order. The model drives the execution loop instead of hard-coded chains.

🧠 Memory

Mechanisms to persist conversation state: ConversationBufferMemory, ConversationSummaryMemory. Gives agents statefulness.

4 LangChain vs LangGraph — The Critical Distinction

LangChain is for linear, directed pipelines. Data flows one direction: input → transform → output. Think of your standard RAG chain: query → retrieve → augment → generate.

LangGraph is for cyclic, stateful workflows. Nodes can loop back. State persists across steps. An agent can retrieve, decide it needs more info, retrieve again, then respond. This is what powers real agentic RAG — and your planned Agentic RAG phase of MyApp lives here.

⚠️
Common Confusion
LangGraph is not a visual flowchart tool despite the word "graph." It's a programming model where you define nodes (functions) and edges (transitions), and LangGraph manages state and routing between them. You write Python, not draw boxes.
Module 1 · Lesson 1
LCEL: The Pipe Operator
LangChain Expression Language (LCEL) is the single most important concept in modern LangChain. It's what lets you compose complex pipelines out of simple pieces using the | operator — like Unix pipes but for AI.
💻 Concept + Code
⏱ ~15 min
1 What LCEL Is

LCEL is the standard way to build chains in LangChain. It was introduced in v0.1 and became the only recommended approach in v1.0+. The old LLMChain, SequentialChain, AgentExecutor etc. moved to langchain-classic — don't use them in new code.

The key insight: every component in LangChain implements the Runnable interface. A Runnable has three methods: .invoke() (single call), .stream() (streaming), and .batch() (parallel calls). Because everything is a Runnable, they can be piped together with |.

LCEL Pipe Composition
PromptTemplate
dict → messages
|
Runnable pipe
ChatAnthropic
messages → AIMessage
|
Runnable pipe
StrOutputParser
AIMessage → str
Each | passes the output of the left Runnable as input to the right Runnable
2 Core LCEL Patterns
python lcel_basics.py 
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
import os

# ─── 1. Build each component ───────────────────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an expert expert assistant for your market."),
    ("human", "Explain {product} your domain in 3 bullet points for {audience}."),
])

model = ChatAnthropic(
    model="claude-3-5-sonnet-20241022",
    api_key=os.getenv("ANTHROPIC_API_KEY")
)

parser = StrOutputParser()

# ─── 2. Compose with pipe operator ─────────────────────────────
# This builds a Runnable chain — nothing runs yet
chain = prompt | model | parser

# ─── 3. Invoke ─────────────────────────────────────────────────
# Single call — runs the full chain
result = chain.invoke({
    "product": "price plan",
    "audience": "first-time buyers"
})
print(result)  # Plain string output

# ─── 4. Stream — token by token output ─────────────────────────
# Identical chain, just call .stream() instead
for chunk in chain.stream({
    "product": "life",
    "audience": "new customers"
}):
    print(chunk, end="", flush=True)

# ─── 5. Batch — parallel calls ──────────────────────────────────
# Runs all inputs concurrently (respects rate limits)
results = chain.batch([
    {"product": "support", "audience": "long-term users"},
    {"product": "travel", "audience": "frequent users"},
    {"product": "asset", "audience": "subscribers"},
])
⚡
invoke vs stream vs batch
.invoke() is for single synchronous calls. .stream() streams tokens progressively — perfect for chat UIs (your MyApp Customer Portal chatbot needs this). .batch() processes multiple inputs concurrently with controlled parallelism — great for document processing pipelines.
3 Async Support — Built In

Every Runnable also has async variants: .ainvoke(), .astream(), .abatch(). Since your MyApp is built on FastAPI with async endpoints, this is exactly what you'll use in production.

python fastapi_integration.py 
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

app = FastAPI()
chain = ChatPromptTemplate.from_template("Answer: {question}") | \
        ChatAnthropic(model="claude-3-5-sonnet-20241022") | \
        StrOutputParser()

# Streaming endpoint — perfect for MyApp Customer Portal chatbot
@app.get("/chat/stream")
async def chat_stream(question: str):
    async def generate():
        async for chunk in chain.astream({"question": question}):
            yield chunk
    return StreamingResponse(generate(), media_type="text/plain")

Prompt Templates
Prompt templates are the input layer of every chain. They separate the structure of your prompt (defined once) from the values (supplied at runtime). This lesson covers every template type you'll use in production.
💻 Code-heavy
⏱ ~15 min
1 PromptTemplate vs ChatPromptTemplate

PromptTemplate produces a single string (for old text-completion models). ChatPromptTemplate produces a list of messages with roles (system, human, assistant) — this is what you need for Claude and all modern chat models.

python prompt_templates.py 
from langchain_core.prompts import (
    ChatPromptTemplate,
    PromptTemplate,
    MessagesPlaceholder,
    HumanMessagePromptTemplate,
    SystemMessagePromptTemplate,
)
from langchain_core.messages import HumanMessage, SystemMessage

# ─── Method 1: from_messages (most common) ───────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an data analyst. Be concise."),
    ("human", "Assess risk for record: {record_details}"),
])

# Invoke to see what gets sent to the model
messages = prompt.invoke({"record_details": "40yo, non-smoker, product"})
print(messages)
# → ChatPromptValue with [SystemMessage, HumanMessage]

# ─── Method 2: Including conversation history ────────────────────
# MessagesPlaceholder injects a variable-length message list
chat_prompt = ChatPromptTemplate.from_messages([
    ("system", "You are an AI assistant. Context: {context}"),
    MessagesPlaceholder(variable_name="chat_history"),  # ← history injected here
    ("human", "{question}"),
])

# This is exactly what your MyApp Developer Chatbot needs:
messages = chat_prompt.invoke({
    "context": "Record document: spec_2024.pdf",
    "chat_history": [               # ← previous turns
        HumanMessage(content="What is my threshold?"),
        AIMessage(content="Your threshold is $500 for at-fault requests."),
    ],
    "question": "What about non-fault requests?",
})
2 Few-Shot Prompting

For tasks like intent classification (your Haiku-based classifier in MyApp), few-shot examples dramatically improve accuracy. LangChain has a FewShotChatMessagePromptTemplate that formats examples cleanly.

python few_shot.py 
from langchain_core.prompts import (
    FewShotChatMessagePromptTemplate,
    ChatPromptTemplate,
)

# Define example input/output pairs
examples = [
    {"input": "What does my record cover?",  "output": "COVERAGE_INQUIRY"},
    {"input": "How do I file a request?",         "output": "REQUEST_PROCESS"},
    {"input": "When does my subscription renew?",    "output": "RENEWAL_INFO"},
    {"input": "I had an accident yesterday",     "output": "INCIDENT_REPORT"},
]

# Template for each example pair
example_prompt = ChatPromptTemplate.from_messages([
    ("human", "{input}"),
    ("ai", "{output}"),
])

# Build the few-shot template
few_shot_prompt = FewShotChatMessagePromptTemplate(
    example_prompt=example_prompt,
    examples=examples,
)

# Wrap in final prompt
intent_classifier_prompt = ChatPromptTemplate.from_messages([
    ("system", "Classify user intent. Respond with only the label."),
    few_shot_prompt,                    # ← injects all examples
    ("human", "{user_message}"),
])
🔗
App Connection
Your Claude Haiku intent classifier in MyApp maps perfectly to this pattern. Replace your manual few-shot string formatting with FewShotChatMessagePromptTemplate and you get cleaner, version-controlled prompt management.
3 Partial Prompts — Pre-fill Variables

Sometimes you want to lock in some template variables at build time and leave others for runtime. .partial() does this:

2 PydanticOutputParser — The Most Powerful

This is the parser you'll use most in production. It instructs the model to output JSON matching a Pydantic schema, then validates and deserialises it. If the model deviates, it raises a validation error you can handle.

python structured_output.py 
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_anthropic import ChatAnthropic
from pydantic import BaseModel, Field
from typing import Optional, Literal

# ─── Define your data schema with Pydantic ──────────────────────
class RequestAssessment(BaseModel):
    request_id: str = Field(description="Unique request identifier")
    risk_level: Literal["LOW", "MEDIUM", "HIGH"] = Field(
        description="Risk classification"
    )
    estimated_payout: float = Field(description="Estimated payout in USD")
    requires_investigation: bool
    notes: Optional[str] = Field(default=None)

# ─── Create parser and extract format instructions ──────────────
parser = PydanticOutputParser(pydantic_object=RequestAssessment)

# The parser generates format instructions to inject into your prompt
print(parser.get_format_instructions())
# → "Return a JSON object with fields: request_id (string), risk_level..."

# ─── Inject format instructions into the prompt ─────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a request assessor. {format_instructions}"),
    ("human", "Assess this request: {request_description}"),
]).partial(format_instructions=parser.get_format_instructions())

model = ChatAnthropic(model="claude-3-5-sonnet-20241022")

# ─── Build chain ────────────────────────────────────────────────
request_chain = prompt | model | parser

# ─── Invoke — returns a RequestAssessment Pydantic object ─────────
result: RequestAssessment = request_chain.invoke({
    "request_description": "Motor collision at Main Street, front bumper damage, $4,500 repair estimate."
})

print(result.risk_level)           # → "LOW"
print(result.estimated_payout)     # → 4500.0
print(result.requires_investigation) # → False
3 with_structured_output — The Modern Way

For models that natively support tool/function calling (like Claude 3.5 Sonnet), there's an even cleaner approach: .with_structured_output(). This uses the model's native JSON mode instead of prompt engineering, which is more reliable.

python structured_output_v2.py 
# Modern approach — uses Claude's native tool calling for reliable JSON
structured_model = ChatAnthropic(
    model="claude-3-5-sonnet-20241022"
).with_structured_output(RequestAssessment)

# No need to inject format instructions into the prompt
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a request assessor."),
    ("human", "Assess this request: {request_description}"),
])

# Chain with structured model — parser is implicit
chain = prompt | structured_model

result: RequestAssessment = chain.invoke({
    "request_description": "Water damage from pipe burst, estimated $12,000."
})
# Guaranteed to return a valid RequestAssessment or raise ValidationError
🏆
Best Practice
Prefer .with_structured_output() over PydanticOutputParser when your model supports it. It's more reliable because it uses the model's native function-calling constraint rather than prompt instructions that the model can deviate from.
Module 1 · Lesson 4
Runnables & Composition
The real power of LCEL comes when you compose complex pipelines with RunnablePassthrough, RunnableLambda, and RunnableParallel. This is where chains start feeling like real orchestration.
💻 Intermediate
⏱ ~18 min
1 The Three Key Runnables

RunnablePassthrough

Passes input through unchanged. Used to include the original question alongside retrieved context in RAG chains.

RunnableLambda

Wraps any Python function as a Runnable so it can participate in a chain. Your glue layer.

RunnableParallel

Runs multiple Runnables in parallel and returns a dict of results. Essential for RAG where you retrieve and pass the question simultaneously.

2 RunnableParallel — The RAG Pattern

The most important use of RunnableParallel is in RAG chains. You need to pass both the retrieved context AND the original question to the prompt. This is how:

python runnable_parallel.py 
from langchain_core.runnables import (
    RunnablePassthrough,
    RunnableParallel,
    RunnableLambda,
)
from langchain_core.prompts import ChatPromptTemplate
from langchain_anthropic import ChatAnthropic
from langchain_core.output_parsers import StrOutputParser

# Assume we have a retriever (covered in Module 2)
# retriever = vectorstore.as_retriever()

# ─── Format retrieved documents into a single string ────────────
def format_docs(docs):
    return "\n\n".join(doc.page_content for doc in docs)

# ─── RAG Prompt ─────────────────────────────────────────────────
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer using only the context below.\n\nContext:\n{context}"),
    ("human", "{question}"),
])

model = ChatAnthropic(model="claude-3-5-sonnet-20241022")

# ─── The RAG chain with RunnableParallel ────────────────────────
# This runs retriever AND passthrough in parallel, producing:
# {"context": "retrieved docs...", "question": "original question"}
rag_chain = (
    RunnableParallel({
        "context": retriever | RunnableLambda(format_docs),  # run retriever, format output
        "question": RunnablePassthrough(),                    # pass input unchanged
    })
    | rag_prompt
    | model
    | StrOutputParser()
)

# ─── Shorthand dict syntax (equivalent, more readable) ──────────
rag_chain_v2 = (
    {
        "context": retriever | RunnableLambda(format_docs),
        "question": RunnablePassthrough(),
    }
    | rag_prompt
    | model
    | StrOutputParser()
)

# Input: just the question string
answer = rag_chain.invoke("What is the threshold for price plan coverage?")
3 RunnableLambda — Custom Logic in Chains
python runnable_lambda.py 
from langchain_core.runnables import RunnableLambda

# ─── Wrap any Python function as a Runnable ──────────────────────
# Example: Presidio PII masking before sending to the LLM
def mask_pii(text: str) -> str:
    # Your Microsoft Presidio logic here
    return text.replace("NRIC", "[REDACTED_ID]")

def add_metadata(output: str) -> dict:
    return {"response": output, "source": "MyApp", "version": "3.0"}

# ─── Compose into a pipeline ─────────────────────────────────────
secure_chain = (
    RunnableLambda(mask_pii)           # Step 1: strip PII
    | ChatPromptTemplate.from_template("Answer: {input}")  
    | ChatAnthropic(model="claude-3-haiku-20240307")
    | StrOutputParser()
    | RunnableLambda(add_metadata)     # Step 5: attach metadata
)

# Shorthand: @chain decorator (cleaner syntax)
from langchain_core.runnables import chain

@chain
def my_custom_step(input: str) -> str:
    # Any logic here — validated by type hints
    return input.upper()

pipe = my_custom_step | model | StrOutputParser()
🔗
MyApp PII Gate
Your Microsoft Presidio PII security gates in MyApp become RunnableLambda steps inserted into the chain before any LLM call. This keeps your security logic composable and testable independently of the model call.

2 Text Splitters — Choosing the Right Strategy
⚠️
Chunking Strategy Matters More Than People Think
Poor chunking = poor retrieval = poor answers. Too large and chunks carry irrelevant noise. Too small and you lose context. Overlapping chunks help maintain context at boundaries. Your MyApp's retrieval quality lives or dies by how you split reference documents.
python splitters.py 
from langchain_text_splitters import (
    RecursiveCharacterTextSplitter,  # ← Use this 90% of the time
    TokenTextSplitter,               # Splits by token count
    MarkdownHeaderTextSplitter,      # Structure-aware for .md
    HTMLHeaderTextSplitter,          # Structure-aware for HTML
    SentenceTransformersTokenTextSplitter,  # For embedding models
)

# ─── RecursiveCharacterTextSplitter — the gold standard ─────────
# Tries to split on: paragraphs → sentences → words → chars
# This preserves semantic coherence as much as possible
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,        # target chunk size in characters
    chunk_overlap=200,      # overlap between consecutive chunks

1 The ReAct Loop

ReAct stands for Reason + Act. The LLM alternates between two modes: reasoning about what to do next (Thought), and taking an action by calling a tool (Act). The tool result (Observation) is fed back to the model, which reasons again. This loop continues until the model decides it has enough info to give a Final Answer.

✅ FINAL ANSWER

2 Tool Design Best Practices
🏆
The Golden Rules of Tool Design
1. Descriptions are prompts. The model reads your tool description to decide when to call it. Write it like a system prompt — tell the model exactly when to use this tool vs another.

2. Narrow > broad. One tool per action. Don't create a "get_record_info" that does 5 things. The model decides which to call; give it clear distinct choices.

3. Always handle errors gracefully. Return error information as a string, not raise exceptions — the agent can reason about error responses and retry.

4. Type hints are mandatory. LangChain generates the JSON schema for the model from your type hints. Missing types = broken tools.

StateGraph Fundamentals
LangGraph 1.1 (March 2026) extends LangChain with a graph execution model. Instead of linear chains, you define nodes (functions) and edges (transitions), giving you full control over cyclic, conditional, and parallel workflows. This is the foundation of Agentic RAG.
🔥 Advanced
⏱ ~22 min
1 Core Concepts: State, Nodes, Edges

State

A TypedDict that is the shared memory of the graph. Every node reads from it and writes to it. It's the "clipboard" that every step can access.

Nodes

Python functions that take the state as input and return a partial state update. They contain your actual business logic — LLM calls, tool calls, database queries.

Edges

Define which node runs after another. Regular edges always go A→B. Conditional edges route based on state — this enables branching and loops.

2 Your First StateGraph
python state_graph_basics.py 
from langgraph.graph import StateGraph, END, START
from langgraph.checkpoint.memory import InMemorySaver
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import BaseMessage, HumanMessage
from typing import TypedDict, Annotated
import operator

# ─── 1. Define State ─────────────────────────────────────────────
# Annotated[list, operator.add] means: when updating, APPEND not replace
class MyAppState(TypedDict):
    messages: Annotated[list[BaseMessage], operator.add]  # conversation history
    user_type: str           # "customer" or "developer"
    retrieved_docs: list     # RAG results
    intent: str              # classified intent

model = ChatAnthropic(model="claude-3-5-sonnet-20241022")
haiku = ChatAnthropic(model="claude-3-haiku-20240307")

# ─── 2. Define Nodes ─────────────────────────────────────────────
# Each node: takes state → returns partial state update
def classify_intent(state: MyAppState) -> dict:
    """Use Haiku to classify the user's intent."""
    last_msg = state["messages"][-1].content
    
    response = haiku.invoke([
        HumanMessage(content=f"Classify intent as REQUESTS, COVERAGE, RENEWAL, or OTHER: {last_msg}")
    ])
    
    intent = response.content.strip().upper()
    if intent not in ["REQUESTS", "COVERAGE", "RENEWAL"]:
        intent = "OTHER"
    
    return {"intent": intent}  # only updates "intent" field in state

def retrieve_documents(state: MyAppState) -> dict:
    """RAG retrieval step."""
    query = state["messages"][-1].content
    docs = retriever.invoke(query)
    return {"retrieved_docs": docs}

def generate_response(state: MyAppState) -> dict:
    """Generate final response using context."""
    context = "\n\n".join(d.page_content for d in state["retrieved_docs"])
    
    response = model.invoke(
        state["messages"] + [
            HumanMessage(content=f"Context:\n{context}")
        ]
    )
    # Annotated list with operator.add means this APPENDS to messages
    return {"messages": [response]}

def handle_requests(state: MyAppState) -> dict:
    """Specialised requests handling node."""
    response = model.invoke(state["messages"] + [
        HumanMessage(content="Assist with this request. Ask for request number if not provided.")
    ])
    return {"messages": [response]}

# ─── 3. Build the Graph ───────────────────────────────────────────
graph = StateGraph(MyAppState)

# Add nodes
graph.add_node("classify", classify_intent)
graph.add_node("retrieve", retrieve_documents)
graph.add_node("generate", generate_response)
graph.add_node("requests_handler", handle_requests)

# Set entry point
graph.add_edge(START, "classify")

# ─── 4. Conditional Routing ──────────────────────────────────────
def route_by_intent(state: MyAppState) -> str:
    """Returns the name of the next node based on intent."""
    if state["intent"] == "REQUESTS":
        return "requests_handler"
    else:
        return "retrieve"   # coverage, renewal, other → RAG

graph.add_conditional_edges(
    source="classify",         # from this node
    path=route_by_intent,       # call this function to decide next node
    path_map={                  # map function output → node names
        "requests_handler": "requests_handler",
        "retrieve": "retrieve",
    }
)

# Regular edges
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)
graph.add_edge("requests_handler", END)

# ─── 5. Compile and run ───────────────────────────────────────────
checkpointer = InMemorySaver()
app = graph.compile(checkpointer=checkpointer)

config = {"configurable": {"thread_id": "customer-session-1"}}
result = app.invoke(
    {
        "messages": [HumanMessage(content="How do I file a request for my product accident?")],
        "user_type": "customer",
        "retrieved_docs": [],
        "intent": "",
    },
    config=config
)

Conditional Edges & Agentic Loops
The defining feature of LangGraph over chains is cycles. A node can route back to a previous node — enabling retry logic, self-refinement, multi-step tool use, and CRAG (Corrective RAG) patterns. This is where the real power lives.

1 CRAG Pattern: Corrective RAG in LangGraph

Your MyApp roadmap includes CRAG. Here's how it maps to LangGraph: after retrieval, a grading node checks if the docs are actually relevant. If not, it loops back to retrieve differently (web search, different query). Only when docs are good does it proceed to generation.

CRAG Flow in LangGraph
retrieve
grade_documents
relevant? → generate
not relevant? → rewrite_query
if rewrite
rewrite_query
loops back ↑
generate
END
python crag_langgraph.py 
from langgraph.graph import StateGraph, END, START
from typing import TypedDict, Annotated, List
from langchain_core.documents import Document
import operator

class CRAGState(TypedDict):
    question: str
    documents: List[Document]
    generation: str
    rewrite_count: int         # prevent infinite loops

# ─── Node: Retrieve ──────────────────────────────────────────────
def retrieve(state: CRAGState) -> dict:
    print(f"Retrieving for: {state['question']}")
    docs = retriever.invoke(state["question"])
    return {"documents": docs}

# ─── Node: Grade Documents ────────────────────────────────────────
def grade_documents(state: CRAGState) -> dict:
    print("Grading document relevance...")
    question = state["question"]
    docs = state["documents"]
    
    # Use LLM to check relevance
    grade_prompt = ChatPromptTemplate.from_messages([
        ("system", "Respond only 'yes' or 'no': Is this document relevant to the question?"),
        ("human", "Question: {question}\nDocument: {document}")
    ])
    grader = grade_prompt | haiku | StrOutputParser()
    
    relevant_docs = []
    for doc in docs:
        score = grader.invoke({"question": question, "document": doc.page_content})
        if "yes" in score.lower():
            relevant_docs.append(doc)
    
    print(f"Relevant: {len(relevant_docs)}/{len(docs)}")
    return {"documents": relevant_docs}

# ─── Node: Rewrite Query ──────────────────────────────────────────
def rewrite_query(state: CRAGState) -> dict:
    print("Rewriting query...")
    rewrite_prompt = ChatPromptTemplate.from_messages([
        ("system", "Rewrite this query to be clearer and more specific for document retrieval."),
        ("human", "{question}")
    ])
    rewriter = rewrite_prompt | haiku | StrOutputParser()
    new_question = rewriter.invoke({"question": state["question"]})
    return {"question": new_question, "rewrite_count": state["rewrite_count"] + 1}

# ─── Node: Generate ───────────────────────────────────────────────
def generate(state: CRAGState) -> dict:
    context = "\n\n".join(d.page_content for d in state["documents"])
    if not context:
        return {"generation": "I couldn't find relevant information in the reference documents."}
    
    gen_prompt = ChatPromptTemplate.from_messages([
        ("system", "Answer based on this context: {context}"),
        ("human", "{question}")
    ])
    chain = gen_prompt | sonnet | StrOutputParser()
    generation = chain.invoke({"context": context, "question": state["question"]})
    return {"generation": generation}

# ─── Conditional Edge Logic ───────────────────────────────────────
def decide_next_step(state: CRAGState) -> str:
    # If we have relevant docs → generate
    # If no relevant docs AND we haven't rewritten too many times → rewrite
    # Safety valve: after 2 rewrites, generate anyway (or graceful fail)
    if state["documents"] or state["rewrite_count"] >= 2:
        return "generate"
    return "rewrite"

# ─── Build CRAG Graph ─────────────────────────────────────────────
crag = StateGraph(CRAGState)
crag.add_node("retrieve", retrieve)
crag.add_node("grade", grade_documents)
crag.add_node("rewrite", rewrite_query)
crag.add_node("generate", generate)

crag.add_edge(START, "retrieve")
crag.add_edge("retrieve", "grade")
crag.add_conditional_edges("grade", decide_next_step, {
    "generate": "generate",
    "rewrite": "rewrite",    # ← cycles back!
})
crag.add_edge("rewrite", "retrieve")   # ← and retrieve again
crag.add_edge("generate", END)

crag_app = crag.compile()
result = crag_app.invoke({
    "question": "What documents do I need for a water damage request?",
    "documents": [],
    "generation": "",
    "rewrite_count": 0
})

1 The Supervisor Pattern

One orchestrator LLM decides which specialised agent (worker) to call next. Workers complete their task and report back. The supervisor continues routing until the task is done. This is the most common multi-agent pattern in production.

from langgraph.graph import StateGraph, END, START
from langchain.agents (langgraph.prebuilt deprecated in v1.0) import create_react_agent
from langchain_anthropic import ChatAnthropic
from langchain_core.messages import BaseMessage, HumanMessage
from typing import TypedDict, Annotated, Literal
import operator

model = ChatAnthropic(model="claude-3-5-sonnet-20241022")

# ─── 1. Create specialised worker agents ─────────────────────────
# Each worker has its own tools and system prompt

data_agent = create_agent(
    model=model,
    tools=[get_record_details, search_policies],
    state_modifier="You are the Data Specialist. You ONLY answer questions about record coverage, terms, and conditions."
)

ops_agent = create_agent(
    model=model,
    tools=[get_request_status, submit_request],
    state_modifier="You are the Operations Specialist. You ONLY handle request submissions, status checks, and requests procedures."
)

pricing_agent = create_agent(
    model=model,
    tools=[calculate_price, get_exchange_rate],
    state_modifier="You are the Pricing Specialist. You ONLY handle price calculations and pricing queries."
)

# ─── 2. Shared state ─────────────────────────────────────────────
class SupervisorState(TypedDict):
    messages: Annotated[list[BaseMessage], operator.add]
    next_agent: str  # which agent to call next

# ─── 3. Supervisor node ──────────────────────────────────────────
workers = ["data_agent", "ops_agent", "pricing_agent", "FINISH"]

supervisor_prompt = ChatPromptTemplate.from_messages([
    ("system", """You are an MyApp supervisor routing requests to specialists.
    
Workers available:
- data_agent: record coverage, terms, exclusions
- ops_agent: request submission, status, procedures  
- pricing_agent: price calculation, pricing
- FINISH: when all parts of the question are answered

Respond with just the worker name to call next. No explanation."""),
    MessagesPlaceholder(variable_name="messages"),
    ("human", "Who should handle this next? Workers: {workers}")
]).partial(workers=", ".join(workers))

def supervisor(state: SupervisorState) -> dict:
    response = model.invoke(
        supervisor_prompt.format_messages(messages=state["messages"])
    )
    next_worker = response.content.strip()
    if next_worker not in workers:
        next_worker = "FINISH"
    return {"next_agent": next_worker}

# ─── 4. Worker node wrappers ─────────────────────────────────────
def run_data_agent(state: SupervisorState) -> dict:
    result = data_agent.invoke({"messages": state["messages"]})
    return {"messages": [result["messages"][-1]]}

def run_ops_agent(state: SupervisorState) -> dict:
    result = ops_agent.invoke({"messages": state["messages"]})
    return {"messages": [result["messages"][-1]]}

def run_pricing_agent(state: SupervisorState) -> dict:
    result = pricing_agent.invoke({"messages": state["messages"]})
    return {"messages": [result["messages"][-1]]}

# ─── 5. Build supervisor graph ────────────────────────────────────
supervisor_graph = StateGraph(SupervisorState)
supervisor_graph.add_node("supervisor", supervisor)
supervisor_graph.add_node("data_agent", run_data_agent)
supervisor_graph.add_node("ops_agent", run_ops_agent)
supervisor_graph.add_node("pricing_agent", run_pricing_agent)

supervisor_graph.add_edge(START, "supervisor")

# Supervisor routes to worker OR ends
supervisor_graph.add_conditional_edges(
    "supervisor",
    lambda x: x["next_agent"],
    {
        "data_agent": "data_agent",
        "ops_agent": "ops_agent",
        "pricing_agent": "pricing_agent",
        "FINISH": END
    }
)

# All workers report back to supervisor
for agent_node in ["data_agent", "ops_agent", "pricing_agent"]:
    supervisor_graph.add_edge(agent_node, "supervisor")

multi_agent_app = supervisor_graph.compile(checkpointer=InMemorySaver())

# ─── 6. Run — complex multi-domain question ──────────────────────
result = multi_agent_app.invoke(
    {
        "messages": [HumanMessage(
            content="I had an accident. How do I file a request on REC-2024-001, and will my price go up next year?"
        )],
        "next_agent": ""
    },
    config={"configurable": {"thread_id": "multi-agent-session-1"}}
)
# Supervisor calls: ops_agent → pricing_agent → FINISH
print(result["messages"][-1].content)
