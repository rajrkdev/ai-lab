WordPiece algorithm — step by step

Choose a word to see exactly how the greedy longest-match algorithm splits it into subword tokens.
Select a word to trace
Algorithm pseudocode
function wordpiece(word, vocab): tokens = [] start = 0 while start < len(word): end = len(word) # try longest first while start < end: substr = word[start:end] if start > 0: substr = "##" + substr # continuation if substr in vocab: tokens.append(substr) break end -= 1 # shrink until found if start == end: return ["[UNK]"] # no match possible start = end return tokens
Why greedy longest-match?
Always tries the longest possible substring first, then shrinks until a vocab match is found.

Maximises information per token — fewer pieces per word = more context fits in the 512-token window.

"insurance" → 1 token (whole word in vocab)
"insurability" → 3 tokens (splits at morpheme boundaries)

Vocabulary structure

BERT's vocabulary has exactly 30,522 tokens. Every token has a unique integer ID. The vocabulary is fixed after pre-training.
Special tokens (reserved IDs)
[PAD]ID 0 — padding to fixed length
[UNK]ID 100 — unknown / unresolvable
[CLS]ID 101 — sentence start / classifier
[SEP]ID 102 — sentence boundary
[MASK]ID 103 — MLM masking token
Vocabulary composition
The ## prefix is not stored in the vocab entry itself — "##ing" is stored as "##ing". The prefix signals to the model that this piece continues the previous token with no whitespace. All tokens including ## pieces are equally valid vocab entries with their own embedding vector.

Text normalisation pipeline

Before WordPiece runs, BERT applies 5 normalisation steps. Each step changes the text in a specific way.

bert-base-uncased vs cased
uncased: lowercases + strips accents. Smaller effective vocab. Better for tasks where case doesn't matter (most NLU tasks).

cased: preserves case. Essential for NER (London ≠ london) and tasks where capitalisation is a signal.
Unicode normalisation (NFC)
The same character can be encoded multiple ways in Unicode.

"é" = U+00E9 (single code point)
"é" = U+0065 + U+0301 (e + combining acute)

NFC normalisation collapses these to the canonical single form before any processing.

Truncation, padding, and the 512-token limit

BERT's position embeddings only go up to 512. Everything beyond is truncated. Everything shorter must be padded. This has direct consequences for RAG chunking.

Chunking strategy for RAG pipelines

For production RAG: use chunk_size=180 tokens with 20-token overlap. At ~100 tokens per sentence this gives ~1.8 sentences per chunk with context continuity. Voyage AI (voyage-2) supports up to 4096 tokens — you can use much larger chunks, which is one of its key advantages over all-MiniLM.

Out-of-vocabulary handling

WordPiece handles most unknown words through subword splitting. True [UNK] only occurs when even individual characters are not in the vocabulary.
OOV word examples — click any
When does [UNK] appear?
Almost never for English text. The 30k vocab covers virtually all English characters and common subwords.

[UNK] appears for:
• Non-Latin scripts (Chinese, Arabic, Cyrillic) in uncased BERT
• Rare unicode symbols (✓, ⚡, emoji)
• Extremely long strings with no valid subwords
WordPiece vs BPE vs Unigram
WordPiece (BERT): likelihood-based. Trained to maximise LM probability of training data.

BPE (GPT, RoBERTa): frequency-based. Merges most frequent pairs iteratively.

Unigram (T5, Albert): probabilistic. Removes tokens that least reduce vocabulary likelihood.
