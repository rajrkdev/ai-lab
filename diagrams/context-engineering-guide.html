The Central Principle

Every file you create for Claude — CLAUDE.md, settings.json, slash commands, agents — is just one way of shaping what goes into the context window. Context Engineering is the strategy. The files are the tactics.

⚙️
settings.json
.claude/settings.json

This is your permission and capability context. It tells Claude what tools it's allowed to use autonomously vs. what requires your approval. Proper settings = Claude that acts, not just talks.

⚙️ settings.json pattern
{
  "model": "claude-opus-4-6",
  "permissions": {
    "allow": [
      "Bash(dotnet:*)",   // run dotnet cmds
      "Write(tests/**)",  // write test files
      "Read(**/*.cs)"    // read any C# file
    ],
    "deny": [
      "Write(*.env)",     // NEVER touch env
      "Bash(rm -rf:*)"   // no dangerous cmds
    ]
  }
}

Slash Commands
.claude/commands/review.md · test-gen.md · fix-bug.md

Slash commands are pre-written context bundles you fire with one word. Typing /review injects a fully formed role, checklist, and output format into the context window instantly — no typing required.

⚡ .claude/commands/review.md

Subagents
.claude/agents/qa-agent.md · jira-agent.md · review-agent.md

Subagents are isolated context environments. Each one is a Claude instance with its own system context, separate from your main conversation. This prevents context pollution — your QA agent stays laser-focused on tests and never gets confused by production code.

🤖 .claude/agents/qa-agent.md
## Identity
QA specialist. ONLY test-related tasks.
Never write production code.

## Isolated Context Rules
- data-testid selectors only
- POM pattern always
- TestDataBuilder for all data
- No Thread.Sleep()

## Input → Output Contract
IN:  Feature description / Jira ticket
OUT: TC-001 format + Playwright C# code
Isolated Context Specialist Role No Pollution
📏
Rules (Hard Constraints)
Defined in CLAUDE.md under "Hard Rules" or "Never Do"

Rules are negative context — they tell Claude what NOT to do. Without explicit rules, Claude defaults to common patterns (like Entity Framework, or using float for money). Rules override those defaults every single time.

📏 Hard Rules block in CLAUDE.md
## Hard Rules — NEVER Break

# Async rules
- NEVER use .Result, .Wait(), GetAwaiter()
- ALWAYS async/await end-to-end

# Money rules (insurance domain)
- ALWAYS decimal for amounts
- NEVER float or double for money

# Privacy rules (APAC compliance)
- NEVER log PII: names, NRICs, policy numbers
- ALWAYS use structured logging with IDs only
Negative Context Domain Safety Override Defaults
🎯
Skills
Patterns + Examples in CLAUDE.md or command files

Skills are example context — the most powerful form of context. Instead of describing what you want, you show Claude a working example. Claude pattern-matches to it. One good example beats three paragraphs of instructions.

🎯 Skill via example in CLAUDE.md

Context Engineering Decisions — FAQ

CLAUDE.md: Information that is ALWAYS true about you, your stack, and your domain. Identity, hard rules, coding standards, and domain constants (decimal for money, etc.). If it's true on every task, every day — it belongs here.

Slash commands: Context that is true for a specific TYPE of task but not all tasks. A review checklist only matters when reviewing. A test-gen template only matters when generating tests. Don't pollute the main context with task-specific instructions.

Rule of thumb: Would I want this in context even when I'm just asking Claude to explain a concept? If no → slash command. If yes → CLAUDE.md.

Use a slash command when you want Claude to do something specific within your current conversation context, and you want it to have access to everything you've discussed so far.

Use a subagent when you want a completely clean-slate Claude instance that is focused ONLY on one specialist job and should NOT be influenced by your conversation history or current code context. Example: your QA agent should not know about the implementation details of the service it's testing — it should only know the requirements and test patterns.

The key insight: subagents provide context isolation, not just specialization. That isolation prevents drift and hallucination from irrelevant context accumulating over a long session.

Signs of context bloat: Claude starts making mistakes it didn't make earlier in the session, response quality drops, it contradicts earlier good output, or it starts combining patterns incorrectly.

When you see this: start a new session. Don't try to correct within the same context window — the noise has accumulated too much. Your CLAUDE.md will re-establish all the important context instantly on the fresh session.

This is why CLAUDE.md is so important — it means a fresh session costs you nothing. You don't lose your context; you just reload it cleanly.

Level 1 — Rule only: "Always use decimal for monetary amounts." Claude knows what to do but may not know why.

Level 2 — Rule + reason: "Always use decimal for monetary amounts — float/double cause rounding errors that fail APAC regulatory precision requirements." Claude now applies this judgment in ambiguous cases too.

Level 3 — Rule + reason + example: Add a code snippet showing the correct pattern. This is the most powerful form. Claude will pattern-match precisely to your example rather than interpreting the rule.

For insurance domain, always use Level 3 for rules around money, dates, PII, and concurrent operations — these are where generic patterns cause real production bugs.

Self-healing is itself a context engineering problem. The context you give the test system determines how it responds when a selector breaks.

In your tests/CLAUDE.md and qa-agent.md, define the fallback hierarchy as an explicit rule: "1. data-testid → 2. aria-label → 3. visible text (log warning)". This becomes the agent's decision tree whenever it generates or repairs a selector.

You can also add a "self-healing protocol" section: when a selector fails at runtime, the agent re-reads the DOM context (tool result) and re-applies the selector priority rules from its context to find the element. The context is what makes the healing intelligent rather than just random retries.

Context Engineering Cheatsheet — Print This

📄 CLAUDE.md — Do This

✓Stack, identity, domain rules — global always-true context
✓Hard rules (async/await, decimal, no PII in logs)
✓One working code example for each core pattern
✓Update it every time Claude gets something wrong
✗Don't add task-specific instructions (those go in commands)
✗Don't make it so long Claude loses the key rules

⚙️ settings.json — Do This

✓Allow: dotnet, git, playwright, read all C# files
✓Restrict writes to src/ and tests/ only
✓Deny: rm -rf, write to .env, write to production configs
✗Don't give blanket Write(**) — too risky
✗Don't deny read access — Claude needs to read to understand

⚡ Commands — Do This

✓One command per task type: /review, /test-gen, /fix-bug
✓Include role + checklist + output format in every command
✓Reference CLAUDE.md rules ("apply CLAUDE.md standards")
✗Don't duplicate what's already in CLAUDE.md
✗Don't make commands so long they bloat the context

🤖 Agents — Do This

✓One specialist per agent — QA, Jira, Review, Doc
✓Define explicit Input → Output contract in every agent
✓Add "Never do" constraints to keep agent in its lane
✗Don't create one "super agent" that does everything
✗Don't let agents share context — defeats the isolation point

📏 Rules — Do This

✓Phrase rules as "ALWAYS X" or "NEVER Y" — be absolute
✓Add the reason: "NEVER float — APAC regulatory precision"
✓Add an example for every critical rule
✗Don't use vague rules: "prefer async" → Claude will ignore it
✗Don't add 50 rules — prioritize your top 10 hard ones

🎯 Prompts — Do This

✓RICE: Role, Information, Constraints, Example — always
✓Paste only the relevant code, not entire files
✓End with "tell me if anything is unclear before writing code"
✗Don't paste the whole codebase — wastes context budget
✗Don't re-explain what's already in CLAUDE.md
