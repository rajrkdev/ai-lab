What is PageIndex?

Traditional Vector RAG
⚠ Similarity ≠ Relevance. Hard chunking destroys table and section context. Cannot follow cross-references.
PageIndex Reasoning RAG
✓ 98.7% accuracy on FinanceBench. Full structure preserved. Explainable reasoning trace per query.
Core Paradigm Shift

PageIndex treats retrieval as a reasoning problem, not a similarity problem. Instead of "which chunk is nearest to this query in embedding space?", it asks: "given this document's structure, where would a human expert navigate to find this answer?"

📐 Tree Node Schema (JSON)
{
  "node_id":     "0006",         // unique ID — used by retriever to fetch raw page content
  "title":       "Exclusions",  // LLM-generated section title
  "start_index": 21,             // start page number (inclusive)
  "end_index":   28,             // end page number (inclusive)
  "summary":     "Describes conditions under which claims are NOT paid...",
  "metadata":    {},             // arbitrary key-value pairs (policy_type, date, etc.)
  "sub_nodes": [                 // recursive array of child sections
    {
      "node_id":     "0007",
      "title":       "Flood Exclusions",
      "start_index": 22,
      "end_index":   24,
      "summary":     "Surface water, storm surge, sewer backup are excluded..."
    }
  ]
}
🎮 Inspired by Monte Carlo Tree Search — The same algorithm behind AlphaGo. PageIndex's LLM does not scan every node exhaustively — it strategically explores promising branches, backtracks when content is insufficient, and commits once it has enough context. Depth-first with heuristic pruning guided by section summaries.
